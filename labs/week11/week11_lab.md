## Review: what changes from Week 10?

In Week 10 you solved one large optimization problem offline to generate a minimum-time trajectory.

In Week 11 you will instead solve a **small problem repeatedly online**:

1. read the current state
2. project it onto the reference
3. extract a finite preview horizon
4. solve a tracking problem over that horizon
5. apply only the **first** control input
6. repeat at the next time step

This is the core idea of **receding-horizon control**.

The optimizer predicts over the next few steps, but only the first command is used. The horizon then shifts forward and the whole problem is solved again from the new measured state.

---

## Tracking state and control

For the current Week 11 scaffold, the tracking MPC uses the path-domain state

$$
x = [t, r, V, \beta, w_r, e, \Delta\phi]^T
$$

with control

$$
u = [\delta, T_r]^T
$$

where:

- $t$ is time carried as a state while integrating with respect to $s$
- $r$ is yaw rate
- $V$ is speed magnitude
- $\beta$ is sideslip angle
- $w_r$ is rear wheel speed
- $e$ is lateral deviation from the reference
- $\Delta\phi$ is the velocity-heading error relative to the reference yaw
- $\delta$ is steering angle
- $T_r$ is rear wheel torque

The horizon is built on a fixed spatial grid

$$
s_k = s_0 + k\,\Delta s
$$

so $s$ itself is not optimized as a state. Instead, the reference window is sampled over that fixed grid and passed into the MPC.

---

## Frenet tracking dynamics

The controller still uses the Week 10 Fiala bicycle model, but the transcription is now written with respect to path progress $s$ rather than time $t$.

The time-domain geometric relations are

$$
\dot s = \frac{V\cos(\Delta\phi)}{1 - e\,\kappa(s)}
$$

$$
\dot e = V\sin(\Delta\phi)
$$

$$
\dot{\Delta\phi} = \dot\beta + r - \kappa(s)\,\dot s
$$

and the physical states follow the Fiala model through

$$
\dot r,\; \dot V,\; \dot\beta,\; \dot w_r
$$

The path-domain model used by the solver is formed by dividing each time derivative by $\dot s$:

$$
\frac{dx}{ds} = \frac{1}{\dot s}
\begin{bmatrix}
1 \\
\dot r \\
\dot V \\
\dot\beta \\
\dot w_r \\
\dot e \\
\dot{\Delta\phi}
\end{bmatrix}
$$

In the helper scaffold, these dynamics are discretized with implicit Euler in $s$ over the fixed horizon grid.

$$
x_{k+1} = x_k + \Delta s_k\, f_s(x_{k+1}, u_k, \kappa_{k+1})
$$

---

## Constraints in the tracking MPC

The controller should respect at least the following constraints:

1. Track bounds:

$$
e_{\min,k} \le e_k \le e_{\max,k}
$$

2. Steering bounds:

$$
-\delta_{\max} \le \delta_k \le \delta_{\max}
$$

3. Torque bounds:

$$
T_{\min} \le T_{r,k} \le T_{\max}
$$

4. State bounds such as speed, sideslip, and yaw rate:

$$
V_{\min} \le V_k \le V_{\max}
$$

$$
\beta_{\min} \le \beta_k \le \beta_{\max}
$$

$$
r_{\min} \le r_k \le r_{\max}
$$

Time-domain slew-rate bounds can be approximated from the path-domain model, but in the current scaffold the main smoothness term is handled in the objective through control increments.

---

## Tracking cost

The current helper uses state tracking, input tracking, and a smoothness penalty on control increments. In the scaffold defaults, most of these weights are set to zero and the main nonzero tracking term is the speed term, so you can turn on the other penalties gradually while tuning.

Here, $\hat{x}_k$ and $\hat{x}_{ref,k}$ denote the scaled state and scaled reference state, and $\hat{u}_k$ and $\hat{u}_{ref,k}$ denote the scaled control and scaled reference control.

$$
J = \sum_{k=0}^{N-1} \left(\|\hat{x}_k - \hat{x}_{ref,k}\|_Q^2 + \|\hat{u}_k - \hat{u}_{ref,k}\|_{R_u}^2 + \|\hat{u}_k - \hat{u}_{k-1}\|_{R_{\Delta u}}^2 \right) + q_{e,f}(\hat{e}_N - \hat{e}_{ref,N})^2 + q_{d\phi,f}(\hat{d\phi}_N - \hat{d\phi}_{ref,N})^2 + q_V(\hat{V}_N - \hat{V}_{ref,N})^2
$$

The connection between the scalar `weight_*` fields and the matrix notation is simply that the implementation uses diagonal weights. In other words:

- $Q$ corresponds to the diagonal state-tracking weights on $(r, V, \beta, w_r, e, \Delta\phi)$
- $R_u$ corresponds to the diagonal input-tracking weights on $(\delta, T_r)$
- $R_{\Delta u}$ corresponds to the diagonal increment weights on $(\delta, T_r)$

In the current implementation, these diagonals are encoded directly as scalar config fields in `TrackingMPCConfig` inside `week11/mpc_helper.py`:

- state tracking: `weight_yaw_rate`, `weight_speed`, `weight_beta`, `weight_wr`, `weight_e`, `weight_dphi`
- input tracking: `weight_steer`, `weight_torque`
- input increments: `weight_steer_increment`, `weight_torque_increment`
- terminal terms: `terminal_e`, `terminal_dphi`, plus `weight_speed` on terminal speed error

To see exactly how the cost is assembled, look at `FrenetTrackingMPC._build_problem()` in `week11/mpc_helper.py`.


The `reference_mode` choice matters:

- `centerline`: the Frenet frame stays attached to the original track centerline. In this mode, `kappa(s)` is the original track curvature, and the optimized Week 10 `e(s)` and `dphi(s)` remain part of the reference state. This is the most natural mode if you want to say the controller is tracking the original track-based trajectory from Week 10.
- `trajectory`: the Frenet frame is attached to the optimized trajectory itself. In this mode, `kappa(s)` is inferred from the optimized Cartesian trajectory, and the reference lies on its own geometry, so the reference `e(s)` and `dphi(s)` become zero. This is the most natural mode if you want to think of the controller as following the optimized race line directly.

For this lab, use `centerline` as the default mode first. After your first working run, also try `trajectory` and compare what changes.

For this lab, start from the same baseline oval/reference setup as in Week 10:

## Part 3 - Assemble the first closed-loop MPC loop

1. read the current simulator measurement
2. convert that measurement into Frenet coordinates
3. sample a finite reference window ahead of the current $s$
4. solve one MPC problem on that window
5. apply only the first control input
6. repeat at the next simulator step
